# Product

## Register

product

## Users

ผู้ดูแลระบบส่วนกลางใช้หน้าจอ Admin เพื่อจัดการร้าน การเชื่อมต่อ SML ผู้รับ สิทธิ์ ตารางแจ้งเตือน และติดตามปัญหาการทำงาน เจ้าของร้านและผู้บริหารเปิด Dashboard ผ่าน LINE/LIFF เพื่อดูตัวเลขของร้านและรายงานที่ตนได้รับสิทธิ์ โดยใช้งานทั้งบนโทรศัพท์และเดสก์ท็อป

## Product Purpose

Nextstep Dashboard ทำให้ข้อมูลจากรายงาน SML ที่อนุมัติไว้ล่วงหน้าทั้ง 10 รายงานเข้าถึงได้อย่างปลอดภัยและตรวจสอบย้อนกลับได้ ผู้ดูแลต้องตั้งค่าร้านและแก้ปัญหาได้โดยไม่ต้องตีความรหัสเทคนิค ส่วนผู้บริหารต้องเห็นตัวเลขสำคัญ แนวโน้ม งวดเปรียบเทียบ และเวลาที่ข้อมูลถูกสร้างโดยไม่ต้องอ่านตารางดิบก่อน

ความสำเร็จหมายถึงผู้ใช้เลือกผิดร้านหรือผิดช่วงเวลาได้ยาก เข้าใจสถานะสด/เก่า/ล้มเหลวทันที เปิดรายงานบนมือถือได้โดยไม่ล้นจอ และตัวเลขทุกค่ากลับไปหา tenant, report run, period และ generated time ได้

## Brand Personality

สุขุม น่าเชื่อถือ กระชับ ภาษาตรงไปตรงมาแบบผู้เชี่ยวชาญที่รับผิดชอบข้อมูล ใช้สีเพื่อบอกการเลือกและสถานะ ไม่ใช้สีเป็นของตกแต่ง ผู้บริหารควรรู้สึกว่ากำลังใช้เครื่องมือควบคุมธุรกิจ ส่วนผู้ดูแลควรรู้สึกว่าสถานะและทางแก้ชัดเจน

## Anti-references

- ไม่เป็นหน้า SaaS marketing หรือหน้า demo ที่มี card ซ้ำกันเพื่อเติมพื้นที่
- ไม่เป็น custom UI ที่แยกตัวจาก Sakai/PrimeVue หรือใช้ component คนละภาษาในแต่ละหน้า
- ไม่ใช้ purple gradient, glassmorphism, เงาหนัก, card มุมโค้งเกินจำเป็น หรือ motion เพื่อการตกแต่ง
- ไม่แสดง database field, UUID, status code หรือศัพท์อังกฤษเป็นคำหลักเมื่อมีคำธุรกิจภาษาไทยที่ชัดกว่า
- ไม่ทำกราฟที่สวยแต่บิดเบือนข้อมูล เช่น รวมจำนวนสินค้าคนละหน่วย หรือใช้ pie/radar โดยไม่มีเหตุผล

## Design Principles

1. **คำตอบก่อนรายละเอียด**: หน้า Dashboard เริ่มจาก KPI, ช่วงข้อมูล, freshness และกราฟที่ตอบคำถามธุรกิจ ก่อนเปิดตารางรายละเอียดตามต้องการ
2. **สถานะต้องนำไปสู่การแก้ไขได้**: loading, empty, stale, partial, conflict และ error ต้องบอกผลกระทบและการกระทำถัดไป
3. **สิทธิ์เป็นข้อมูลจาก Backend**: เมนูช่วยลดความสับสน แต่ authorization ทุก tenant/report/run บังคับที่ Server เสมอ
4. **ใช้ Sakai ให้เป็นระบบเดียว**: ใช้ app shell, component vocabulary, spacing, typography และ theme token ของ Sakai/PrimeVue ก่อน custom CSS
5. **ตัวเลขต้องอธิบายได้**: แสดงหน่วย งวดเปรียบเทียบ เวลาไทย และ data-quality state; ไม่สร้างข้อสรุปหรือคำแนะนำจาก AI

## Accessibility & Inclusion

เป้าหมาย WCAG 2.2 AA รองรับ keyboard, focus ที่มองเห็นได้, dialog focus trap/restore, screen reader, contrast ที่ผ่านเกณฑ์, reduced motion, zoom/reflow 200–400% และ viewport ตั้งแต่ 320px กราฟทุกชุดต้องมีคำอธิบายและข้อมูลแบบตารางที่เข้าถึงได้โดยไม่พึ่งสีหรือ canvas เพียงอย่างเดียว
